# Functional & Non-Functional Requirements
## Luminary — AI Reading & Learning Companion for Dyslexic Learners

**Status:** Draft v1.0
**Companion to:** PRD-Luminary-Dyslexia-Learning-Companion.md
**Last updated:** June 30, 2026

Requirements are tagged by ID and priority: **P0** (MVP, launch blocker), **P1** (Phase 2), **P2** (Phase 3 / future). Each functional requirement maps back to a user story or feature in the PRD.

---

## 1. Functional Requirements

### 1.1 Onboarding & Profiles

| ID | Requirement | Priority |
|---|---|---|
| FR-1.1 | The system shall allow a parent/guardian to create an account using email or social sign-in. | P0 |
| FR-1.2 | The system shall require verifiable parental consent before creating a profile for a child under 13. | P0 |
| FR-1.3 | The system shall allow a parent to create, edit, and delete one or more child profiles (name, age, approximate reading level). | P0 |
| FR-1.4 | The system shall let a child or parent select reading preferences during setup: font, text size, spacing, color/contrast theme, and TTS voice. | P0 |
| FR-1.5 | The system shall allow preferences to be changed at any time from a settings screen, not just during onboarding. | P0 |
| FR-1.6 | The system shall support multiple child profiles under a single parent account, each with independent settings and progress data. | P1 |

### 1.2 Read-to-Me (TTS + OCR)

| ID | Requirement | Priority |
|---|---|---|
| FR-2.1 | The system shall allow a user to capture a printed page via camera and convert it to digital text using OCR. | P0 |
| FR-2.2 | The system shall allow a user to import text via paste, file upload (PDF/TXT), or URL. | P0 |
| FR-2.3 | The system shall read imported/captured text aloud using text-to-speech. | P0 |
| FR-2.4 | The system shall visually highlight each word on screen in sync with the audio as it is spoken (assisted-reading mode). | P0 |
| FR-2.5 | The system shall allow the user to control playback: play, pause, rewind by sentence, and adjust speed between 0.7x and 1.5x. | P0 |
| FR-2.6 | The system shall offer at least 2 distinct TTS voices at launch, expandable in later phases. | P0 |
| FR-2.7 | The system shall save imported/scanned documents to a personal library for later re-reading. | P0 |
| FR-2.8 | The system shall support a "line focus" reading mode that dims all but 1–3 active lines of text. | P0 |
| FR-2.9 | The system shall expand the voice library to 10+ voices and additional accents. | P1 |
| FR-2.10 | The system shall support translation of selected words for bilingual users. | P2 |

### 1.3 AI Comprehension Support

| ID | Requirement | Priority |
|---|---|---|
| FR-3.1 | The system shall allow a user to tap/select any word to receive a simple, age-appropriate definition generated by the AI model. | P0 |
| FR-3.2 | The system shall allow a user to select a passage and request a simplified rewrite that preserves the original meaning at a lower reading level. | P0 |
| FR-3.3 | The system shall generate a short summary of an imported document or long passage on request. | P0 |
| FR-3.4 | The system shall sound out a tapped word phonetically (syllable breakdown) in addition to TTS playback. | P1 |
| FR-3.5 | The system shall provide a picture/visual aid alongside word definitions for younger or more severe readers. | P1 |
| FR-3.6 | The system shall cache or rate-limit repeated AI calls for the same content to control cost and latency. | P0 |

### 1.4 Oral Reading Practice (Core Differentiator)

| ID | Requirement | Priority |
|---|---|---|
| FR-4.1 | The system shall present the user with a short reading passage appropriate to their reading level. | P0 |
| FR-4.2 | The system shall record and analyze the user's spoken reading using speech recognition. | P0 |
| FR-4.3 | The system shall detect skipped, substituted, or mispronounced words during oral reading. | P0 |
| FR-4.4 | The system shall provide real-time, non-punitive feedback on flagged words (e.g., soft highlight plus correct-pronunciation playback) without interrupting reading flow more than necessary. | P0 |
| FR-4.5 | The system shall require a minimum confidence threshold before flagging a word as mispronounced, to reduce false corrections. | P0 |
| FR-4.6 | The system shall generate an end-of-session summary highlighting specific, genuine wins (not generic praise). | P0 |
| FR-4.7 | The system shall maintain a running list of a user's recurring "tricky words/sounds" across sessions. | P0 |
| FR-4.8 | The system shall allow the user to replay any flagged word's correct pronunciation on demand. | P0 |
| FR-4.9 | The system shall offer spaced-repetition review sessions targeting a user's personal tricky-word list. | P1 |
| FR-4.10 | The system shall adapt passage difficulty based on recent session accuracy (adaptive leveling). | P1 |

### 1.5 Progress & Parent Dashboard

| ID | Requirement | Priority |
|---|---|---|
| FR-5.1 | The system shall generate a weekly summary for parents: minutes practiced, passages completed/read, and recurring tricky words/sounds. | P0 |
| FR-5.2 | The system shall NOT expose a granular, word-by-word error transcript to parents by default. | P0 |
| FR-5.3 | The system shall allow a parent to opt in to more detailed session data if explicitly requested. | P1 |
| FR-5.4 | The system shall display simple, motivating progress indicators to the child (streaks, levels, badges) without framing mistakes punitively. | P0 |
| FR-5.5 | The system shall allow export of progress summaries (e.g., PDF) for sharing with teachers or specialists. | P2 |

### 1.6 Account, Billing & Access

| ID | Requirement | Priority |
|---|---|---|
| FR-6.1 | The system shall offer a free tier including TTS/OCR reading, customization, and line-focus mode. | P0 |
| FR-6.2 | The system shall gate AI tutoring, simplification/summarization, and premium voices behind a paid subscription tier. | P0 |
| FR-6.3 | The system shall support monthly and annual subscription billing. | P0 |
| FR-6.4 | The system shall allow a parent to cancel or downgrade a subscription at any time without contacting support. | P0 |
| FR-6.5 | The system shall display zero third-party advertising at any subscription tier. | P0 |

## 2. Non-Functional Requirements

### 2.1 Performance

| ID | Requirement | Priority |
|---|---|---|
| NFR-1.1 | OCR text extraction from a captured page shall complete within 5 seconds on a standard mobile connection (4G/Wi-Fi). | P0 |
| NFR-1.2 | TTS playback shall begin within 2 seconds of a user pressing play for cached/previously-processed content. | P0 |
| NFR-1.3 | Real-time oral-reading feedback shall be delivered with no more than ~1 second of perceptible lag between a spoken word and any visual flag, to preserve reading flow. | P0 |
| NFR-1.4 | AI-generated definitions/simplifications shall return within 3 seconds for typical passage lengths (under ~300 words). | P0 |
| NFR-1.5 | The application shall remain responsive (no UI freeze) during background AI calls; long operations shall show progressive loading states. | P0 |

### 2.2 Reliability & Availability

| ID | Requirement | Priority |
|---|---|---|
| NFR-2.1 | Core reading features (TTS, OCR, library access, saved preferences) shall function with a documented, graceful offline/degraded mode when connectivity is poor. | P1 |
| NFR-2.2 | The backend AI service layer shall target 99.5% uptime during peak usage hours (after-school/homework hours in primary markets). | P0 |
| NFR-2.3 | The system shall handle AI provider errors/timeouts gracefully with a clear, non-alarming message to the child user (no raw error codes shown to children). | P0 |
| NFR-2.4 | User progress data and saved documents shall be durably persisted with automated backups; data loss tolerance is zero for progress history. | P0 |

### 2.3 Accuracy & Quality (AI-specific)

| ID | Requirement | Priority |
|---|---|---|
| NFR-3.1 | OCR accuracy on real-world printed material (textbooks, worksheets) shall meet a defined minimum word-level accuracy threshold, tracked continuously in production. | P0 |
| NFR-3.2 | False-correction rate in oral-reading feedback (flagging a correctly-read word as wrong) shall be tracked as a core quality metric and kept below an agreed threshold, given its outsized impact on user trust. | P0 |
| NFR-3.3 | AI-simplified text shall be evaluated periodically for meaning preservation (not just reading-level reduction) using sampled human review. | P1 |
| NFR-3.4 | Speech recognition models used for oral-reading analysis shall be evaluated specifically on children's speech samples (not adult-speech benchmarks alone), given known accuracy gaps. | P0 |

### 2.4 Usability & Accessibility

| ID | Requirement | Priority |
|---|---|---|
| NFR-4.1 | All core reading screens shall meet WCAG 2.1 AA contrast and text-resizing standards. | P0 |
| NFR-4.2 | The interface shall support adjustable font, spacing, and color themes as first-class settings, not hidden in nested menus. | P0 |
| NFR-4.3 | The application shall be fully operable via touch and voice input for users with co-occurring motor or attention difficulties. | P1 |
| NFR-4.4 | Session length and UI pacing shall default to short, low-fatigue intervals (10–20 minutes), consistent with recommended dyslexia practice routines. | P0 |
| NFR-4.5 | Error and correction states shall use supportive, shame-free language and visuals, validated with target users (children) before launch. | P0 |
| NFR-4.6 | The app shall be usable by a first-time child user without adult assistance after initial parent-led setup. | P1 |

### 2.5 Privacy, Security & Compliance

| ID | Requirement | Priority |
|---|---|---|
| NFR-5.1 | The system shall comply with COPPA requirements for any user account associated with a child under 13, including verifiable parental consent. | P0 |
| NFR-5.2 | The system shall comply with GDPR/UK GDPR data-minimization and right-to-erasure requirements for users in applicable jurisdictions. | P0 |
| NFR-5.3 | The system shall NOT sell or share user data (reading content, audio, progress data) with third parties for advertising purposes. | P0 |
| NFR-5.4 | The system shall minimize retention of raw oral-reading audio recordings, storing only derived accuracy/progress signals beyond a short processing window by default. | P0 |
| NFR-5.5 | All data in transit and at rest shall be encrypted using industry-standard protocols (TLS 1.2+, AES-256 or equivalent). | P0 |
| NFR-5.6 | The system shall provide a clear, age-appropriate, parent-readable privacy policy and in-app data controls (view/delete child data). | P0 |
| NFR-5.7 | The system shall achieve FERPA-compliant data handling ahead of any school/educator-facing release (Phase 3). | P2 |
| NFR-5.8 | API keys and AI provider credentials shall never be exposed client-side; all AI calls shall be proxied through a secured backend service layer. | P0 |

### 2.6 Scalability & Maintainability

| ID | Requirement | Priority |
|---|---|---|
| NFR-6.1 | The services layer integrating with the AI provider (Gemini) shall be abstracted so the underlying model/provider can be swapped without UI-layer changes. | P1 |
| NFR-6.2 | The system architecture shall support horizontal scaling of AI request handling during peak homework-hour load. | P1 |
| NFR-6.3 | The codebase shall maintain automated test coverage for core reading, OCR, and oral-practice flows to support safe iteration. | P0 |
| NFR-6.4 | Feature flags shall be used to roll out AI-feature changes (especially oral-feedback logic) gradually and reversibly given their trust sensitivity. | P1 |

### 2.7 Compatibility

| ID | Requirement | Priority |
|---|---|---|
| NFR-7.1 | The application shall support modern iOS and Android mobile browsers/devices at launch (camera + microphone access required). | P0 |
| NFR-7.2 | The application shall support major desktop browsers (Chrome, Safari, Edge, Firefox) for the reading/library/dashboard experience. | P0 |
| NFR-7.3 | The application shall degrade gracefully on low-end devices common in the target market (lower-cost Android tablets/phones). | P1 |

## 3. Traceability Notes

- Oral-reading feedback requirements (FR-4.x, NFR-3.2, NFR-3.4) are the highest-risk area identified in research and should receive the earliest prototyping and the most rigorous quality gates before launch.
- Font/typography requirements (FR-1.4, NFR-4.2) are deliberately framed as comfort/preference settings rather than claimed reading-speed interventions, consistent with the evidence reviewed in the PRD.
- Privacy/compliance requirements (Section 2.5) should be treated as launch blockers, not post-launch cleanup, given how heavily competitors lean on trust signals ("ad-free," "no data selling," compliance badges) as a parent purchase driver.
